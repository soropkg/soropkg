# soropkg publish

Publish your contract package to the soropkg registry so others can install it.

```bash
soropkg publish [flags]
```

---

## How publishing works

<Steps>

### Authenticate with GitHub

soropkg uses GitHub OAuth to verify package ownership. On first publish, your browser opens a GitHub auth page. Once authorised, ownership is tied to your GitHub org/username — which must match the `name` prefix in your `soroban.toml`.

### Verify on-chain

soropkg fetches the live WASM from the network addresses declared in your manifest and computes its hash. This hash is stored in the registry alongside your package — ensuring that what you publish matches what is actually deployed.

### Publish to registry

Your manifest, contract specs, and verified WASM hashes are submitted to the registry. The package becomes discoverable via [`soropkg search`](/cli/search) and installable via [`soropkg add`](/cli/add).

</Steps>

---

## Requirements

Before publishing:

1. A valid `soroban.toml` with at least one network's contract addresses
2. GitHub authentication (org/username must match package name prefix)
3. Contracts must be deployed and accessible on the specified network

---

## Flags

| Flag | Description |
|---|---|
| `--mainnet` | Publish mainnet contract addresses (default) |
| `--testnet` | Publish testnet contract addresses |
| `--dry-run` | Validate and verify without actually publishing |
| `--tag <tag>` | Add a release tag (e.g. `stable`, `beta`) |

## Versioning

Use standard [semver](https://semver.org) in your `soroban.toml`. Each published version is immutable — you cannot overwrite an existing version. To release an update, bump the version in your manifest and publish again.

---

<Callout type="warning">
  **Package names are namespaced to your GitHub identity.** You can only publish packages whose `name` prefix matches your authenticated GitHub username or organisation.
</Callout>
