# On-chain contract specs

One of Soroban's most powerful features is that **contract interfaces are stored directly inside the deployed WASM binary** on the blockchain. This means you never need a separate ABI file, and the interface you read is always the canonical, live one.

---

## What is a contract spec?

When you compile a Soroban contract with the `#[contractimpl]` and `#[contracttype]` macros, the Rust SDK automatically encodes a machine-readable spec — called `contractspecv0` — into a custom section of the WASM binary.

This spec describes:

- All **public functions** — their names, argument names, argument types, and return types
- All **custom types** — structs, enums, and union types used by the contract
- All **error codes** — named error variants and their numeric values

---

## How soropkg reads specs

When you run `soropkg inspect` or `soropkg add`, soropkg:

1. Fetches the live WASM binary from the Stellar RPC for the given contract address
2. Parses the `contractspecv0` custom section from the WASM
3. Deserialises the XDR-encoded spec entries
4. Formats them as a human-readable interface (or JSON, or TypeScript types)

This means soropkg always reflects the **current deployed state** of the contract — not a stale cached version.

## Type mapping

Soroban types map to TypeScript as follows:

| Soroban type | TypeScript type |
|---|---|
| `i32`, `u32` | `number` |
| `i64`, `u64`, `i128`, `u128` | `bigint` |
| `bool` | `boolean` |
| `Symbol`, `String` | `string` |
| `Address` | `string` (Stellar address) |
| `Bytes`, `BytesN<N>` | `Buffer` |
| `Vec<T>` | `Array<T>` |
| `Map<K, V>` | `Map<K, V>` |
| `Option<T>` | `T \| null` |
| Custom struct | TypeScript `interface` |
| Custom enum | TypeScript `enum` |
