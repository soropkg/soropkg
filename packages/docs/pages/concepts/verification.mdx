# WASM verification

When a package is published to the soropkg registry, the CLI computes and stores a cryptographic hash of the deployed WASM binary. This hash anchors the registry entry to the actual on-chain contract — making it impossible to publish a package that misrepresents what is deployed.

---

## How it works

When you run `soropkg publish`:

1. soropkg reads your `soroban.toml` and finds your declared contract addresses
2. It fetches the **live WASM binary** from the Stellar RPC for each address
3. It computes the **SHA-256 hash** of the binary
4. The hash is stored in the registry alongside your package metadata
5. On every future `soropkg add` or `soropkg inspect`, the hash can be re-verified against the live chain

---

## Why this matters

Without verification, a malicious actor could publish a package that claims to be `blend-capital/blend` but points to a different contract — a honeypot, a rug, or simply the wrong version.

WASM verification provides a trust anchor:

- **The WASM hash in the registry matches the WASM hash on-chain.**
- If a contract is upgraded after publishing, `soropkg inspect` will reflect the new interface, and a new version must be published to update the registry entry.

---

## Checking a hash manually

You can verify a registry entry yourself with `soropkg inspect --json`:

```bash
soropkg inspect CAQQR5SWBXKIGZKPBZDH3KM5GQ5G... --json | jq .wasmHash
# "8a3fc1d2e7b9f04a5c3e8d1b6f2a9e4c7d0b3f5a8e1c4d7b0a3f6e9c2d5b8a"
```

Then cross-reference with the Stellar explorer:

```bash
# Stellar Laboratory / Lobstr / StellarExpert all show the WASM hash
# for any contract on the Contract Info page
```

---

<Callout type="warning">
  If a contract is **upgraded on-chain** after its package was published, the registry WASM hash will no longer match the live contract. This is expected — it signals that the package needs a new version to be published. The `soropkg inspect` command will always show the **current** live interface regardless.
</Callout>
